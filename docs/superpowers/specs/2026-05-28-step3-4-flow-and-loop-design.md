# v2 Step 3+4 — Flow Table + Closed-Loop Enforcement Design

Status: **proposed** (2026-05-28). Implements build-order steps 3 and 4 of `docs/SPEC_v2.md` in
one combined cycle (step 3's flow table and step 4's rule enforcement are tightly coupled — both
live in the verdict path — so one design / plan / build / silicon validation saves a Vivado
cycle and the redundant brainstorm overhead). Layers on the locked 8 MHz / 32 B link, opcode
sublayer, and runtime-write infrastructure (see [[v2-step1-telemetry-done]],
[[v2-step2-runtime-rules-done]]).

## Goal

**Step 3:** Replace v1.1's 256-bucket undercounting per-source table with a **4096-bucket
fingerprinted** `flow_table.v` that *detects* collisions instead of silently merging.
**Step 4:** Use the rule_store loaded in step 2: a `rule_lookup.v` reads the store on every
classify frame and ORs its match into the verdict, closing the **observe → decide → act** loop
when the Pi polls telemetry and pushes a block-rule for the heaviest talker.

## Architecture

```
spi_slave_rx → header_parser → classifiers (verdict-side)
                                  ├─ bloom_filter         (unchanged)
                                  ├─ flow_table  [NEW step 3, replaces scan_rate]
                                  └─ rule_lookup [NEW step 4, reads rule_store]
                              telemetry (unchanged, step 1)
                              thresholds + rule_store + bloom write port (step 2)
```

Verdict combine: `hit_mask = {rule, rate, port_scan, bloom}` (bit 3 = rule hit, was reserved);
`severity = max(rule.severity, classifier.severity)`; `escalate = any bit | rule.action[2]`.

## Step 3 — `flow_table.v` (replaces `scan_rate.v`)

**Locked decisions:**
- **Key = `src_ip`** — spec field set (`dport_fp`, `dhost_fp`) is per-source bookkeeping. "Flow"
  in the spec is loose; same verdict semantics as v1.1 scan_rate.
- **Bucket hash = `(src_ip * A1) >> 20`** — top 12 bits of low-32 product, 4096 buckets. Same
  multiply-shift family as v1.1.
- **Fingerprint = `(src_ip * A2) >> 12 & 0xFFFF`** — 16 bits from a DIFFERENT multiplier (A2),
  uncorrelated with the bucket index; collision-on-collision rate ~1/65,536.
- **Cell = `{fp:16, epoch:8, pkt_count:14, byte_count:24, syn_count:12, dport_fp:16, dhost_fp:16,
  flags:8}` = 114 bits.** 4096 × 114 b → **13 RAMB36** (4096×9 mode).
- **Replace, don't add alongside.** `scan_rate.v` stays in the tree but `classifiers.v` no
  longer instantiates it; `flow_table.v` is dropped in. Verdict signals (`port_scan_hit`,
  `rate_hit`) keep the same names so verdict frame layout / `verdict_encoder.v` are unchanged.
- **4-phase FSM mirrors scan_rate.v** — phase 0 latch hashes (DSP isolated), phase 1 BRAM read
  latency, phase 2 fp-compare + RMW + latch new counts, phase 3 popcount + threshold compare.
  Same timing-closure shape; the bigger BRAM doesn't add depth.

**Update logic per packet:**

1. Compute `bucket`, `fp`.
2. Read cell at `bucket`.
3. `eff = (stored.epoch == cur_epoch && stored.fp == fp) ? stored_state : ZEROS`.
4. `pkt_count++` (saturating), `byte_count += pkt_size`, on SYN-no-ACK
   `syn_count++` and `dport_fp |= 1<<port_bit` and `dhost_fp |= 1<<host_bit`,
   `flags |= tcp_flags`.
5. Write back `{fp, cur_epoch, new_state}` — collision evicts the previous source's state.
6. `port_scan_hit ← popcount(dport_fp) ≥ port_thresh || popcount(dhost_fp) ≥ host_thresh`;
   `rate_hit ← pkt_count ≥ rate_thresh`.

**Regression watch:** the existing **120/120** silicon golden was generated by v1.1's
256-bucket twin. ~30 distinct sources hashed into 256 buckets via `A1>>24` *might* have had
collisions silently merged. Step 3 evicts them, producing **more accurate** verdicts on those
frames — but if the golden encodes a v1.1 collision, the silicon regression dips below 120/120.
**This is a known-good improvement** — if it happens, regenerate the affected rows from the
new twin and document the divergence.

## Step 4 — `rule_lookup.v` + closed-loop Pi

### HDL

**New module `rule_lookup.v`:** ~30 lines.

```verilog
// reads rule_store at hash(src_ip), checks (stored.src_ip == src_ip) AND
// (stored.epoch == current_rule_epoch). Outputs (match, action:8, severity:4).
module rule_lookup (
  input  wire        clk, rst,
  input  wire [31:0] src_ip,
  input  wire        in_valid,
  input  wire [7:0]  current_rule_epoch,
  output wire [8:0]  rs_r_idx,      // wires to rule_store r_idx
  input  wire [71:0] rs_r_rule,     // from rule_store r_rule (1-cycle latency)
  output reg         match, out_valid,
  output reg  [7:0]  action,
  output reg  [3:0]  severity
);
```

**Lookup hash:** `idx = (src_ip * A1) >> 23 & 0x1FF` — top 9 bits of the low-32 product →
0..511 buckets. **The Pi writes rules at the same index.** Direct-mapped, single-cycle read.

**Decode of `rs_r_rule` (72 b):** unpack to `{src_ip:32, action:8, severity:4(low), epoch:8}`
exactly the layout pinned in `PROTOCOL.md` / `rule_store_model.py`.

**`current_rule_epoch` register:** lives in `thresholds.v` as `threshold_id = 0x03`. Writable
via opcode `0x11` (extending the existing threshold IDs); the Pi increments it to mass-expire
all rules at once. Read via `0x14`.

**Wiring in `nids_top.v`:**

- `thresholds.v` gains a 4th internal register `rule_epoch_r` (default 0 on reset), exposes
  `rule_epoch` tap to `rule_lookup`.
- `rule_lookup.v` instance fed by `classify_now` + `src_ip`. Output drives a new path into
  `classifiers.v`'s verdict combine.
- `rule_store.r_idx` becomes a mux: `(ctrl_now && opcode==0x15) ? Pi_idx : rule_lookup_idx`.
  The two readers never overlap (Pi 0x15 frames don't go through `classify_now`).

**Verdict combine** (in `classifiers.v` or `nids_top.v`):

```
mask        = {rule_match, rate_hit, port_scan_hit, bloom_hit};   // bit 3 is the new rule hit
sev_classif = bloom_hit ? 3 : ((port_scan|rate) ? 2 : 0);
severity    = max(sev_classif, rule_severity);                    // 2-bit max with the 4→2-bit truncation
escalate    = (any classifier bit) | rule_action[2];              // bit 2 of action is "escalate"
```

`verdict_encoder.v` layout unchanged — bit 3 of `hit_mask` was reserved; the verdict frame still
parses identically with `verdict.py`'s existing decoder (it ignores bits 3–7 of mask anyway).
Optional small enhancement: add `"rule"` to `_STAGE_BITS` in `verdict.py` so `decode_verdict`
surfaces a `rule_hit` field — backward compatible.

### Pi-side closed loop (`closed_loop.py`, ~80 lines)

```
loop:
  s = decode_snapshot(query(0x02))
  if s.top1_count >= TRIGGER_THRESH:
      ip = s.top1_key
      idx = ((ip * A1) & MASK32) >> 23 & 0x1FF             # same hash as the FPGA
      rule = {src_ip: ip, action: 0b101, severity: 3, epoch: current_epoch}
      send(encode_rule_write(idx, rule))
      log(f"rule pushed: block {fmt_ip(ip)} @ idx {idx}")
  sleep(POLL_INTERVAL)
```

`TRIGGER_THRESH` is a CLI knob (default e.g. 100 packets/window). `current_epoch` starts at 0;
the script could rotate it (advance + push fresh rules) but for step-4 baseline behavior it
stays constant.

## TDD plan (the part that gates execution)

### Python twins + tests (no HDL build needed for these)

- **`flow_table_model.py`** — bit-exact twin: same bucket/fp hashes, same lazy-epoch reset,
  same eviction-on-fp-mismatch semantics. `update(src_ip, dst_ip, dst_port, proto, tcp_flags,
  pkt_size, frame_count) → (port_scan_hit, rate_hit)`. Must produce identical outputs to
  `scan_rate.py` when no collisions occur (the no-collision case is the v1.1 golden stream's
  expected behavior).
- **`test_flow_table_model.py`** — unit tests: counter saturation, lazy reset, **directed
  collision-evict test** (two sources whose `*A1>>20` collide → second evicts first), no-effect
  of fp-mismatch (eviction doesn't preserve old state).
- **`rule_lookup_model.py`** + tests — lookup hash, match-only-if-src_ip-AND-epoch, decode of
  stored rule fields. Plus end-to-end: write rule for X via rule_store_model, query
  rule_lookup_model for X → match; query for Y (different src_ip same idx by hash collision) →
  no match.
- **`closed_loop_test.py`** — pure-logic test of the policy decision: given a synthetic
  snapshot blob, verify the script picks the right `(idx, rule)` to push.

### HDL + sim (one Vivado sim cycle for the lot)

- **`tb_flow_table.v`** — replays a stream including a known collision (two sources crafted to
  share a bucket but differ in fp), asserts eviction. Plus the v1.1 scan-detection patterns
  (vscan, flood, hscan, boundary) — same expected `port_scan_hit`/`rate_hit` outputs as
  `tb_scan_rate` since the workload has no v1.1-style collisions.
- **`tb_rule_lookup.v`** — drive a few rule_store contents (preloaded directly via the new
  module's r_idx interface, bypassing the Pi-side path), query several src_ips, assert
  match/no-match + action/severity passthrough.
- **`tb_nids_top.v` extension** — one classify frame for a src that has a rule loaded;
  one for a src that doesn't. Assert verdict frame bit 3 of mask reflects rule_match.
- **WNS check** — flow_table has the same 4-phase shape as scan_rate but a wider FP+state RMW.
  Watch the WNS report; pipeline the popcount stage if it tightens.

### Silicon validation

1. **v1.1 regression** — `spi_verdict_check.py` → 120/120 (or surface the v1.1-collision
   divergence honestly if any).
2. **Step 1+2 regressions** — `silicon_telemetry_check.py` + `silicon_runtime_check.py`
   unchanged (those modules weren't touched).
3. **Step 3 directed silicon test** — `silicon_flow_check.py` sends a known stream including
   collision-inducing src_ip pairs; asserts the second source's state is *clean* (not OR'd with
   the first's) via the verdict outputs.
4. **Step 4 closed-loop demo** — start `closed_loop.py` in the background; flood from one src
   IP (via `silicon_telemetry_check`'s STREAM at scale); within ~2 s observe (a) the snapshot's
   top1 spikes, (b) the script pushes a rule, (c) subsequent classify frames for that src come
   back with `rule_hit` set and severity escalated. The whole loop visible in real time.

## Resource impact

- **flow_table.v:** +13 BRAM. (was: scan_rate ~1 BRAM, so net +12.)
- **rule_lookup.v:** zero BRAM (combinational + 1 register stage; reads existing rule_store).
- **thresholds.v** gains a 4th register: still zero BRAM (flops).
- **Chip total after step 3+4:** ~28.5 BRAM (57%). Comfortable headroom.
- DSP: ~+3 for flow_table's hashes (replacing scan_rate's similar count). Net change minimal.

## Scope boundaries (YAGNI)

- **Read-out opcode for `flow_table` state** — the richer fields (`byte_count`, `syn_count`,
  `flags`) are stored but not exposed via SPI. A future opcode `0x16` could surface them
  (mirroring snapshot/HLL readouts) — not in this step.
- **Multi-action rule policies** — the closed-loop script pushes one fixed rule template
  (block + escalate, severity 3). More nuanced policies (rate-limit, alert-only,
  per-protocol) are step 5+ extensions.
- **Rule eviction policy on Pi side** — when index N already holds a different IP's rule, the
  Pi just overwrites. No LRU bookkeeping. Acceptable since the closed loop is heuristic.
- **Step 5** is separate (Pi Zero / Pi 4 traffic generators + a top-level demo orchestrator —
  pure Pi-side, no HDL, no build).

## Open / watch

- **v1.1 regression divergence** (above) — handle when discovered.
- **`current_rule_epoch` initial value** — defaults to 0. The Pi must use the same value when
  writing initial rules. If the Pi runs `closed_loop.py` fresh after a board reset, both start
  at 0 → consistent. A mass-expire is one threshold write (`0x11 id=0x03 value=new_epoch`).
- **WNS** — currently +0.289 ns. The added rule_lookup mux + flow_table fp-compare could
  pressure the verdict combine path. If WNS drops below ~+0.1 ns, pipeline `classifiers.v`'s
  verdict combine into its own register stage.
- **rule_store r_idx mux** — drives both Pi readback (opcode 0x15) and classifier lookup.
  Verify they never overlap in time (they shouldn't, since 0x15 frames aren't classify frames,
  but assert it in the tb).
